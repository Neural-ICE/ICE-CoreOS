# ADR-0004 — Two-domain TPM2/LUKS disk encryption

- **Status**: Accepted (validated on the GB10 test hardware)
- **Date**: 2026-06-29
- **Decider**: Business/Security Owner (human)
- **Related to**: [ADR-0003](ADR-0003-base-and-update-model.md) (bootc foundation, native OTA),
  [ADR-0002](ADR-0002-secure-boot-zero-touch.md) (Secure Boot — PCR 7).

## Context

Need formulated by the decider: *"encrypt as much as possible, at minimum 2 distinct
keys that auto-unlock from the TPM at boot, and for one of the two keys we
hand the recovery key to the device owner at installation (backed up to the USB
+ shown on screen)."*

With the former **Ubuntu + Mender (A/B rootfs)** appliance, encrypting the system
partition was impractical:

1. **Block deployment vs LUKS**: Mender writes a raw image to the inactive
   slot; the LUKS container *is* the block device → you cannot rewrite a
   rootfs into it without recreating the container.
2. **TPM sealing broken on every switch**: slot A and slot B have
   different kernels/initrd → **different PCR** measurements → the TPM refuses to
   unseal after an A↔B switch.

## Why bootc/ostree unblocks encryption

There is **only one physical rootfs**. The multiple OS deployments live as
*content-addressed* hardlinks in `/ostree` **on that same filesystem**. We therefore
encrypt **a single LUKS volume**, unlocked once at boot; `bootc upgrade` writes
the new version **as files into the already-unlocked filesystem**. Encryption
is **transparent** to updates.

## Decision

### Two LUKS2 volumes (maximal encryption)

| Partition | Size | Content | Key |
|---|---|---|---|
| `p1` ESP | 1 GiB | signed EFI binaries (public) | — clear |
| `p2` `/boot` | 1 GiB | signed kernel + initramfs (public) | — clear |
| `p3` **LUKS "system"** | 300 GiB | `/ostree` + `/var` (logs, containers) | **TPM2 PCR 7** + recovery (operator escrow) |
| `p4` **LUKS "data"** | rest (~3.4 TB) | `/var/lib/neural-ice/data` (user data, models) | **TPM2 PCR 7** + **OWNER recovery** (USB + screen) |

Only the ESP and `/boot` stay in the clear: they contain only **Secure Boot
signed** binaries (integrity by signature, no need for confidentiality on
a public kernel). Everything else (OS *and* data) is encrypted at rest.

### TPM sealing = PCR 7 only

Both volumes seal their key to the TPM via `systemd-cryptenroll
--tpm2-device=auto --tpm2-pcrs=7`. **PCR 7 measures the Secure Boot state** (PK/KEK/db +
certificates used), **not the kernel hash**. Since all shipped kernels
are signed by the same key, **PCR 7 stays constant across `bootc upgrade`**
→ auto-unlock survives updates. Disabling Secure Boot or changing the keys
changes PCR 7 → the TPM refuses to unseal → **tamper-evident**.

### Recovery model (escrow)

- **System**: the OS is *reinstallable* from GHCR (nothing irreplaceable) → its
  recovery key is an **operator escrow** (file on the USB).
- **Data**: irreplaceable → its recovery key is **handed to the device owner**: shown on
  screen AND backed up to the USB (`NEURAL-ICE-RECOVERY-<serial>.txt`).

Both recovery keys are generated by `systemd-cryptenroll --recovery-key`
(modhex format, 8 groups, BitLocker type). If the board/TPM is replaced, PCR 7
changes → auto-unlock drops → the recovery key restores access.

### Implementation

Manual partitioning (`sfdisk`) + `bootc install to-filesystem` (documented path
for LUKS), with karg `rd.luks.uuid=luks-<UUID>` +
`rd.luks.options=<UUID>=tpm2-device=auto` for the root. The data volume is mounted
post-boot via `/etc/crypttab` (`tpm2-device=auto,nofail`) + `/etc/fstab`, written
into the deployment's `/etc`. Everything is in the **installer** (not in the public
OS image, which stays minimal and reusable).

## Threat model (accepted by the decider)

TPM auto-unlock protects against **SSD extraction** (a stolen disk then read
elsewhere = unreadable). It **does not protect** against an attacker who **boots
the appliance** on site — by design: autonomous edge, no human to enter
a passphrase at boot. For a stronger need (defense), a
Tang/Clevis network-bound variant of the data remains possible (see options not retained).

## Hardware validation (GB10 test hardware, kernel-64k, 2026-06-29)

Everything tested on a loop file (never the real nvme) with the **real TPM**:

- `systemd-cryptenroll --tpm2-pcrs=7` → **enroll OK** (token `systemd-tpm2`, PCR 7,
  sha256, SRK).
- `--recovery-key` → **OK** (token `systemd-recovery`, modhex 8 groups).
- `systemd-cryptsetup attach … tpm2-device=auto` → **auto-unlock OK** (LUKS2
  aes-xts-plain64) without a passphrase.
- **Bug found + fixed**: `cryptsetup luksKillSlot` refuses to kill slot 0 with
  the key of slot 0 → replaced with `systemd-cryptenroll --wipe-slot=0` (validated:
  leaves TPM + recovery, TPM unlock still OK afterwards).
- `bootc install to-filesystem` on an **encrypted root** → **OK**; the generated BLS
  entry does contain `rd.luks.uuid=luks-… rd.luks.options=…=tpm2-device=auto`.

> Still to validate on the real install: **TPM unlock on first boot**
> (PCR 7 of the live USB environment vs the installed system). The recovery escrow covers
> the case where they would diverge.

## Consequences

- **Initramfs**: the CentOS Stream 10 bootc base already embeds `crypt` + `tpm2-tss`
  + `systemd-cryptsetup` → **no rebuild of the canonical image** needed for
  LUKS/TPM unlock at boot.
- **Installer tools**: everything is in the base (`sfdisk`/`partx`/`cryptsetup`/
  `systemd-cryptenroll`/`mkfs.*`) → **zero added package**.
- **MS-signed shim migration** (ADR-0002): re-signing the shim will change PCR 7 once
  → plan an idempotent service that **re-enrolls the TPM** on first boot if
  the unseal fails.

## Options not retained

- **Data on TPM PCR 7** *(retained)* vs **Tang/Clevis network-bound** (off-site
  theft = dead data, but depends on a server at boot) vs **bound to measured
  application state**. The decider chose **zero-touch TPM PCR 7** for
  both domains (sufficient for GDPR "data at rest", no human
  action).
- **`bootc install to-disk --block-setup tpm2-luks`**: only handles **one** volume
  occupying the whole disk → incompatible with the second "data" domain. Hence
  the manual partitioning + `to-filesystem`.
